# Production-like локальный запуск (RAG API)

## Что нужно для стабильной работы

- **Docker Desktop** (или Linux + Docker) с достаточным объёмом диска под том Postgres.
- **PostgreSQL 16 + pgvector** — образ `pgvector/pgvector:pg16` уже указан в `docker-compose.yml`.
- **Ollama на хосте** (Windows/macOS/Linux): GPU желательна для чата и эмбеддингов; API в контейнере ходит к Ollama по `http://host.docker.internal:11434` (см. `extra_hosts` в compose).
- **Модели в Ollama**:
  - эмбеддинги: `nomic-embed-text` (768 измерений — должно совпадать с миграцией `vector(768)`);
  - чат: `llama3:8b`, `qwen3:8b` или аналог по объёму VRAM/RAM.
- **Redis** в compose опционален для RAG-логики, но включён для health и будущего кэша; `REDIS_URL` по умолчанию указывает на сервис `redis` в сети compose.

Минимальная последовательность:

```bash
docker compose up -d postgres redis rag-api
curl -s http://localhost:8080/health
# Ollama должна быть запущена на хосте
curl -s -X POST http://localhost:8080/seed
```

## Ограничения Ollama

- Скорость и качество зависят от **RAM/VRAM** и выбранной модели.
- Эмбеддинги и чат — **отдельные** модели; при смене эмбеддингов на другую размерность потребуется правка миграции (`vector(N)`) и полная переиндексация.
- Ollama не даёт «облачного» дообучения: знания — только через **ваши файлы** и RAG.

## RAM и ресурсы (ориентиры)

- **PostgreSQL + pgvector**: от ~512 MB для пустой БД; рост с числом чанков и размером индекса HNSW.
- **rag-api (Python)**: ~200–400 MB в покое; пик при `/seed` и `/query`.
- **Ollama**: для `llama3:8b` ориентируйтесь на **8+ GB RAM** под модель; для 30B — существенно больше или квантование.
- Индекс **HNSW** увеличивает размер БД относительно сырых текстов.

## Рекомендуемые модели (практика)

| Роль | Пример | Комментарий |
|------|--------|-------------|
| Эмбеддинги | `nomic-embed-text` | 768-d, хороший баланс для RAG |
| Чат (RU/юр. стиль) | `llama3:8b`, `qwen3:8b` | Проверяйте факты; юр. точность не гарантируется |
| Больший контекст | модели с большим `num_ctx` в Ollama | Настраивается в клиенте/Modelfile отдельно |

## Риски для legal use

- Ответы **не являются** юридическим заключением.
- В режиме **`strict`** при плохом retrieval **LLM не вызывается** (`llm_invoked: false`) — снижается риск выдуманных норм, но ответ может быть отказом «мало материалов».
- При **хорошем** retrieval в `strict` LLM **всё ещё вызывается** — остаётся **остаточный риск** галлюцинаций; системный промпт сужает, но не устраняет полностью. Для максимальной строгости используйте только цитаты из `citations` и сверяйте с первоисточниками.
- **`balanced`** и **`draft`** предполагают большую свободу формулировок — обязательна ручная проверка.

## Проблемы с pgvector / HNSW

- Если при миграции падает создание индекса `USING hnsw`, проверьте версию **pgvector** в образе и логи Postgres.
- Обходной путь: временно закомментировать в `001_rag_schema.sql` блок `CREATE INDEX ... hnsw` и создать индекс вручную после наполнения (например IVFFlat) — см. документацию pgvector.
- После изменения типа индекса выполните `REINDEX` или пересоздайте индекс в окне обслуживания.

## Наблюдаемость

- `GET /health` — готовность Postgres, Ollama, Redis.
- `GET /stats` — счётчики документов/чанков, последний ingestion.
- Таблица `query_logs` — история запросов (для отладки; при production-like нагрузке продумайте ротацию/очистку).

## Следующие шаги (продукт)
