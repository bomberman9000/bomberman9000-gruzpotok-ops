## Открыто, но не блокирует релиз (Open, non-blocking)

| Тема | Описание |
|------|----------|
| **Кейс 1 — residual variance** | По одному и тому же запросу (Самара — Казань, ~3 т, без фиксированного кузова) модель даёт **разные вилки** между прогонами (наблюдались диапазоны порядка 70–110 … 150–250 тыс. руб.). Это **не** баг retrieval и не поломка формы ответа, а **дисперсия генерации** и «уход» от якоря. |
| **Редкие хвосты формулировок** | Смешение меток «Ставка:» + лексика «ориентир» в одном ответе; тавтология «Нужны уточнения… нужны уточнения» после санитайзера. **Polishing**, не rescue. |

## Следующий backlog (один маленький трек)

**Задача-заготовка:** `pricing_variance_case1`

1. Заморозить текущий прогон как **baseline** (уже есть дампы).
2. При необходимости метрики: **10+ повторов** кейса 1 → медиана, min/max, ширина вилки относительно ближайшего якоря в контексте.
3. **Вариант A (предпочтительно):** ещё одна микро-итерация **только prompt** — жёстче привязка к retrieved anchor (не уводить вилку дальше заданного допуска от якоря); **одна** лексика цены на ответ («ставка» **или** «ориентир», не оба смысла).
4. **Вариант B (позже, production-hardening):** лёгкая **постобработка чисел** — если извлечена одна вилка, проверка ширины и мягкое сжатие к допустимому коридору относительно anchor (anti-chaos leash).

Рекомендация: сначала **A**, постобработку чисел — когда понадобится жёсткая стабилизация под прод.

### Границы следующего трека (обязательно)

Следующая работа — **отдельный трек**, не продолжение «раскопок» текущего цикла.

| В зоне трека | Вне трека (не трогать без новой причины) |
|--------------|-------------------------------------------|
| Сужение **case 1 variance** (один и тот же запрос чеклиста: Самара — Казань, ~3 т, кузов не выбран) | Инфраструктура, Docker, routing, backend API |
| `logistics.txt` и при необходимости **узкая** постобработка ответа logistics (только если согласовано) | **Retrieval** (top-k, rerank, фильтры), индексация, корпус, `seed` |
| Метрики: **before / after** — одинаковый сценарий, N повторов, сравнение с `docs/evals/case1_repeat5.json` как **baseline** | **`legal.txt`**, claim-review, JSON-легал-пайплайн |
| Дампы: `case1_repeat10_before.json` / `case1_repeat10_after.json` (или аналог) | Повторный аудит `RETRIEVAL_CASE2_PRICING_AUDIT.md` без новой проблемы |

**Остановочная точка:** текущий цикл зафиксирован как **нормальный baseline** — не «вроде стало лучше», а явный срез в `FINAL_ITERATION_SUMMARY.md` + артефакты в `docs/evals/`. Следующий шаг — только осознанный **case 1 variance** с измеримым до/после.
