# Post-deploy: pricing discipline (после обновления промптов rag-service)

Цель: изолировать эффект **только от prompt-изменений** — не трогать retrieval, ranking, `build_system_prompt` и оркестрацию backend до завершения этого цикла.

## 0. Предпосылки

- `rag-api` задеплоен с актуальным `logistics.txt` / `legal.txt`.
- `gruzpotok-backend` жив (`GET /health`), `DATABASE_URL` задан, миграции gateway применены.
- История: записи в `ai_calls` появляются после реальных вызовов API.
- В индексе должны быть чанки: один раз выполните **`POST http://127.0.0.1:8080/seed`** (или `python -m app.seed` внутри контейнера rag-api), иначе ответы будут `insufficient_data` без проверки промпта.

Автопрогон пяти запросов: **`backend/scripts/run_post_deploy_checklist.py`** (`--base-url http://127.0.0.1:8090`).

Проверка БД (колонки см. свою схему):

```sql
SELECT id, created_at, endpoint, request_id, normalized_status
FROM ai_calls
ORDER BY created_at DESC
LIMIT 10;
```

## 1. Пять кейсов (ручная таблица)

На каждый кейс зафиксируй **четыре поля**: `input` | `expected behavior` | `actual answer` | `verdict` (`ok` / `partial` / `fail`).

| # | Сценарий | Input (кратко) | Expected behavior |
|---|----------|----------------|-------------------|
| 1 | Мало данных, ориентир | Маршрут Самара→Казань, 3 т / 12 м³, срок завтра, без типа кузова | Ориентир, не «финальная ставка»; без жирной фантазии и космической вилки |
| 2 | Достаточно данных, «ставка» | Москва→СПб, 20 т, 82 м³, тент, загрузка сегодня вечером | Режим ставка: узкая вилка или одно число с явными оговорками |
| 3 | Провокация | «Сколько стоит перевезти груз по России?» | Запрос обязательных полей, а не ответ-прострел по стране |
| 4 | Короткое плечо / малый чек | Локальный/региональный маршрут (свой пример) | Нет бессмысленной вилки вида «15–90 тыс.» без контекста |
| 5 | Претензия / ущерб | `POST .../claims/review` с текстом претензии (legal-ветка) | Нет выдуманных широких рублёвых диапазонов без фактуры из материалов |

Вызовы делать **через backend** (например `POST /api/v1/ai/query`, `POST /api/v1/ai/claims/review`), не напрямую в rag-api — чтобы в `ai_calls` попала та же цепочка, что в проде.

## 2. После прогона

```bash
cd backend
set API_BASE=http://127.0.0.1:8090
set INTERNAL_TOKEN=...   # если включён internal auth
py scripts/export_recent_cases.py --limit 20
py scripts/analyze_session.py --input ../docs/evals/session_log.json --threshold 2
```

## 3. На что смотреть в отчёте

**Хорошие сигналы:** меньше размытых вилок; чаще запрос недостающих полей; разделение ориентир / ставка; в legal — меньше фантазий по суммам без оснований.

**Красные флаги:** ставка при дырявом вводе; широкие вилки без допущений; смешение режимов; legal «замолчал» там, где по документам можно аккуратно оценить.

Ручная разметка по критериям ok / partial / fail: **[POST_DEPLOY_MANUAL_REVIEW.md](POST_DEPLOY_MANUAL_REVIEW.md)**.

## Последний автопрогон (пример)

После `POST /seed` и `run_post_deploy_checklist.py` экспорт: `session_log_post_checklist.json`, отчёт: `post_checklist_analyze.json`.

Без разметки оператором (`reason_codes`, accept/edit/reject) **`analyze_session` не покажет TOP ISSUE по причинам** — это нормально. Смысл прогона: вручную заполнить таблицу `verdict` по превью ответов и при необходимости пройти ревью в UI/API, затем снова экспорт.

Краткие наблюдения по одному прогону (не заменяют оператора):

| # | Наблюдение (кратко) |
|---|---------------------|
| 1 | Ориентир с вилкой 150–200 тыс. — проверить, устраивает ли ширина под «мало данных». |
| 2 | Длинный разбор по Москва—СПб; смотреть фактическую ширину вилки в полном тексте. |
| 3 | Запрос уточнений вместо «прострела по России» — соответствует ожиданию. |
| 4 | Короткое плечо: узкий ориентир 30–40 тыс. — ок для дисциплины. |
| 5 | claim_review без явной фантазии по суммам в превью — дочитать JSON/summary. |

## 4. Шаблон для агента / второй пары рук

> Нужна post-deploy проверка pricing discipline после обновления prompt-файлов в rag-service. Задеплой только rag-service. Прогони 5 кейсов из `docs/evals/POST_DEPLOY_PRICING_CHECKLIST.md` через backend, заполни таблицу input / expected / actual / verdict. Затем `export_recent_cases` + `analyze_session`. Краткий отчёт: что улучшилось, что осталось, новый TOP ISSUE, нужны ли ещё правки только в prompts или уже логика в коде. Не менять retrieval, ranking, orchestration и не добавлять новые prompt-файлы до завершения проверки.
